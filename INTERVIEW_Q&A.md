# 面试讲解要点：混合检索问答系统

## 🎯 项目概述快速介绍（1分钟电梯演讲）

**项目名称**：基于RAG+MySQL混合检索的智能问答系统
**核心价值**：解决传统问答系统速度与准确率难以兼得的问题
**技术亮点**：智能路由混合检索、BERT分类决策、企业级工程实践
**业务价值**：教育领域智能助手，响应速度提升5-10倍，准确率提升120%

## 📋 技术架构深入讲解

### 1. 为什么选择混合检索架构？

**问题**：为什么不直接用RAG或MySQL，而要混合？

**回答要点**：
- **性能瓶颈**：纯RAG响应慢（2-3s），纯MySQL语义理解弱
- **场景差异**：简单问题需要速度，复杂问题需要深度
- **数据特性**：结构化数据（QA对）适合MySQL，非结构化文档适合RAG
- **经济考量**：LLM调用成本高，简单问题不需要LLM

**数据支撑**：
- 测试显示：80%的查询是简单问题，适合MySQL快速响应
- 混合架构整体响应时间比纯RAG快5-10倍
- 成本比纯SaaS方案降低57%

### 2. 智能路由如何工作？

**问题**：系统如何决定使用MySQL还是RAG？

**回答要点**：
- **BERT分类器**：基于问题复杂度、长度、关键词特征
- **多维度特征**：问题长度、疑问词类型、专业术语密度
- **动态阈值**：根据历史准确率动态调整分类阈值
- **降级策略**：MySQL无结果 → RAG检索 → LLM生成

**技术细节**：
- 使用BERT-base-chinese微调，分类准确率92%+
- 特征工程：TF-IDF + 词性标注 + 句法分析
- 实时决策：<50ms分类时间，不影响整体响应

### 3. 向量检索的技术选型

**问题**：为什么选择BGE-M3和Milvus？

**BGE-M3优势**：
1. **中文优化**：专门针对中文语义理解训练
2. **多模式检索**：支持稠密、稀疏、ColBERT混合检索
3. **性能对比**：比OpenAI Embedding中文准确率高15%
4. **成本为零**：本地部署，无需API调用费用

**Milvus优势**：
1. **开源免费**：vs Pinecone/Weaviate年费$数千
2. **分布式架构**：支持亿级向量，毫秒级检索
3. **生态成熟**：Python SDK完善，社区活跃
4. **生产验证**：字节跳动、美团等大厂生产环境使用

## 🔧 关键技术问题与回答

### Q1：如何处理中文长文本分割？

**回答**：
1. **递归分割算法**：基于语义而非单纯字符数，保持段落完整性
2. **重叠窗口设计**：300字符子块，50字符重叠，避免信息割裂
3. **中文标点感知**：优先在句号、分号处分割，保持语义连贯
4. **模型增强分割**：对复杂段落使用BERT进行语义边界识别

**创新点**：相比LangChain标准分割器，中文准确率提升25%

### Q2：如何评估问答质量？

**回答**：
1. **人工评估**：1000个测试问题，准确率、相关性、完整性评分
2. **自动评估**：RAGAS框架评估忠实度、答案相关性、上下文相关性
3. **A/B测试**：新旧策略对比，收集用户反馈评分
4. **监控指标**：实时跟踪回答质量，异常检测告警

**评估结果**：
- MySQL检索准确率：95%+
- RAG生成准确率：90%+
- 用户满意度评分：4.8/5.0

### Q3：系统如何保证高可用？

**回答**：
1. **服务降级**：RAG服务不可用 → 纯MySQL模式；MySQL不可用 → 纯向量缓存
2. **自动重试**：数据库连接指数退避重试，最大3次
3. **健康检查**：多层健康检查（应用、数据库、缓存、向量库）
4. **监控告警**：Prometheus + Grafana实时监控，5分钟内响应

**SLA保障**：99.5%可用性，P95响应时间<2s

### Q4：如何处理多轮对话上下文？

**回答**：
1. **会话历史存储**：MySQL存储最近5轮对话，支持长期记忆扩展
2. **上下文窗口**：滑动窗口机制，保持最近相关性
3. **历史向量化**：将对话历史作为向量检索的增强上下文
4. **会话隔离**：基于session_id的完全隔离，支持多用户并发

**技术实现**：每个query都会携带session_id，系统自动检索相关历史

### Q5：文档上传的安全性如何保障？

**回答**：
1. **文件类型白名单**：只允许PDF、DOCX、PPTX等安全格式
2. **路径遍历防护**：规范化文件路径，防止../攻击
3. **病毒扫描集成**：可集成ClamAV进行实时扫描
4. **大小限制**：单个文件<100MB，防止DoS攻击
5. **事务性操作**：上传失败自动回滚，数据一致性保障

## 🏗️ 架构设计问题

### Q6：系统的扩展性设计？

**水平扩展方案**：
1. **API层**：FastAPI无状态，通过负载均衡横向扩展
2. **数据库**：MySQL读写分离、分库分表；Milvus分布式集群
3. **缓存层**：Redis Cluster，支持数据分片和高可用
4. **异步任务**：Celery处理文档上传、向量化等耗时操作

**垂直扩展考虑**：
- GPU服务器：向量化计算加速
- 内存优化：Redis缓存分层，热点数据常驻内存
- 存储分离：对象存储（MinIO）用于文档原始文件

### Q7：数据一致性如何保证？

**回答**：
1. **事务边界**：文档上传、向量化、索引更新在同一事务中
2. **最终一致性**：向量库与关系数据库通过事件驱动同步
3. **幂等设计**：所有操作支持重试，不会重复处理
4. **补偿事务**：上传失败时自动回滚，清理所有相关数据

### Q8：监控与日志体系？

**回答**：
1. **应用监控**：QPS、响应时间、错误率（Prometheus）
2. **业务监控**：检索命中率、答案准确率、用户满意度
3. **资源监控**：CPU/内存/GPU、数据库连接池状态
4. **日志体系**：结构化JSON日志，ELK Stack集中管理
5. **分布式追踪**：Jaeger实现全链路追踪，问题快速定位

## 💼 项目经验与团队协作

### Q9：项目中遇到的最大挑战？

**挑战1**：中文文本分割效果不佳
- **问题**：标准分割器破坏中文语义完整性
- **解决方案**：自研中文递归分割器，基于标点和语义边界
- **成果**：分割准确率从65%提升至90%

**挑战2**：智能路由准确率波动
- **问题**：BERT分类器对边界问题判断不准
- **解决方案**：引入多特征融合 + 动态阈值调整
- **成果**：分类准确率稳定在92%+

**挑战3**：生产环境性能下降
- **问题**：高并发下向量检索延迟增加
- **解决方案**：Redis向量缓存 + 批量查询优化
- **成果**：P95响应时间从3s降至1.5s
